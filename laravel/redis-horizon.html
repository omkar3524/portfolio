Laravel Redis, Queues & Horizon

Running Horizon under Supervisor, inspecting running jobs, clearing queues, and finding queue names.

What is Horizon?

Laravel Horizon is a dashboard and code-driven configuration layer for your Redis-backed queues. It gives you real-time metrics (throughput, runtime, failures), retry tooling, and queue-worker auto-balancing — all configured in config/horizon.php. It only works with the redis queue driver.

1. Installation & Setup

# require the package
composer require laravel/horizon

# publish config + assets (creates config/horizon.php)
php artisan horizon:install

# make sure your queue connection is redis
# .env
QUEUE_CONNECTION=redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379

Dashboard is served at /horizon. Lock it down in app/Providers/HorizonServiceProvider.php via the gate() method.

2. Horizon Artisan Commands

# start the Horizon master process (spawns all workers)
php artisan horizon

# check supervisor/worker status: running | paused | inactive
php artisan horizon:status

# pause / resume job processing
php artisan horizon:pause
php artisan horizon:continue

# gracefully terminate — finishes the current jobs, then exits.
# Run this on every deploy so Supervisor restarts a fresh process
# with your new code.
php artisan horizon:terminate

# clear all jobs from a queue (default queue if omitted)
php artisan horizon:clear
php artisan horizon:clear --queue=emails

# delete a single failed job from the dashboard
php artisan horizon:forget {id}

# store a metrics snapshot (schedule this every 5 min)
php artisan horizon:snapshot

Schedule the snapshot in routes/console.php (or the Kernel):

Schedule::command('horizon:snapshot')->everyFiveMinutes();

3. Keeping Horizon Alive with Supervisor

The php artisan horizon process must always be running. Supervisor restarts it automatically if it crashes or after a deploy. Create /etc/supervisor/conf.d/horizon.conf:

[program:horizon]
process_name=%(program_name)s
command=php /var/www/your-app/artisan horizon
directory=/var/www/your-app
autostart=true
autorestart=true
user=www-data
redirect_stderr=true
stdout_logfile=/var/www/your-app/storage/logs/horizon.log
stopwaitsecs=3600

Important: stopwaitsecs must be longer than your longest-running job so Supervisor doesn't kill a job mid-flight.

Then load and manage the program:

# re-read config files for new/changed programs
sudo supervisorctl reread

# apply the changes (start newly added programs)
sudo supervisorctl update

# manual control
sudo supervisorctl start horizon
sudo supervisorctl stop horizon
sudo supervisorctl restart horizon

# check status of everything Supervisor manages
sudo supervisorctl status

# wildcard restart (useful when process_name is numbered)
sudo supervisorctl restart horizon:*

4. General Queue Commands

These work with any queue driver. With Horizon you normally let it manage workers, but these are essential for debugging:

# process jobs (Horizon does this for you under the hood)
php artisan queue:work redis --queue=high,default

# tell all workers to restart after the current job (deploys)
php artisan queue:restart

# list failed jobs
php artisan queue:failed

# retry failed jobs
php artisan queue:retry all
php artisan queue:retry {uuid}

# delete failed jobs
php artisan queue:forget {uuid}   # one
php artisan queue:flush           # all

# alert when a queue backs up beyond a threshold
php artisan queue:monitor redis:default,redis:emails --max=100

5. How to Clear a Queue

# clear PENDING jobs from a queue (Artisan)
php artisan queue:clear redis --queue=default
php artisan queue:clear redis --queue=emails

# or via Horizon
php artisan horizon:clear --queue=default

# nuke directly in Redis (last resort)
redis-cli DEL queues:default
redis-cli DEL queues:default:delayed
redis-cli DEL queues:default:reserved
⚠️ Heads up

queue:clear removes pending jobs but not jobs already reserved by a running worker. queue:flush only clears the failed jobs table. Deleting Redis keys directly is irreversible.

6. How to Find Running / Pending Jobs in Redis

Jobs live in Redis under keys named queues:{name}. Inspect them with redis-cli:

redis-cli

# list every queue key that exists
KEYS queues:*

# PENDING jobs (a Redis LIST) — count and contents
LLEN  queues:default            # how many are waiting
LRANGE queues:default 0 -1      # show the raw job payloads

# RESERVED jobs = currently being processed (a sorted set)
ZCARD  queues:default:reserved          # how many are running
ZRANGE queues:default:reserved 0 -1     # show them

# DELAYED jobs = scheduled for the future (a sorted set)
ZCARD  queues:default:delayed
ZRANGE queues:default:delayed 0 -1 WITHSCORES   # score = run-at timestamp

Quick mental model of the three states:

  • Pending → queues:default (LIST) — waiting to be picked up.
  • Reserved → queues:default:reserved (ZSET) — a worker is running it now.
  • Delayed → queues:default:delayed (ZSET) — released/retried, due later.

For a friendlier view, the /horizon dashboard shows pending, running, completed and failed jobs in real time.

7. How to Find the Queue Name

A job's queue name comes from one of these (in order of precedence):

// 1. Set explicitly when dispatching
ProcessPodcast::dispatch($podcast)->onQueue('emails');

// 2. Set on the job class itself
class ProcessPodcast implements ShouldQueue
{
    public $queue = 'emails';
    // or in the constructor:
    public function __construct() {
        $this->onQueue('emails');
    }
}

The default name (when none is set) is defined in config/queue.php:

'redis' => [
    'driver' => 'redis',
    'connection' => 'default',
    'queue' => env('REDIS_QUEUE', 'default'),  // <-- default queue name
    'retry_after' => 90,
    'block_for' => null,
],

To discover which queue names actually exist at runtime, list the Redis keys:

# every queue name currently in use shows up here
redis-cli KEYS 'queues:*'

# example output:
#   queues:default
#   queues:emails
#   queues:emails:reserved
#   queues:notifications:delayed

The queues Horizon supervises are also listed under each supervisor's queue key in config/horizon.php.

8. Finding the Correct Redis Key (the Prefix)

Why KEYS queues:* returns nothing

Laravel prepends a prefix to every Redis key. So a queue stored as queues:default in your code actually lives at laravel_database_queues:default in Redis. If you query the bare name you'll find nothing — you have to include the prefix.

Find the exact prefix your app uses with Tinker:

php artisan tinker
// the prefix Laravel adds to every Redis key
config('database.redis.options.prefix');
// => "laravel_database_"

// build the real key for a queue
config('database.redis.options.prefix') . 'queues:default';
// => "laravel_database_queues:default"

The prefix is defined in config/database.php and defaults to your app name + _database_:

'redis' => [
    'options' => [
        'prefix' => env('REDIS_PREFIX', Str::slug(
            env('APP_NAME', 'laravel'), '_'
        ) . '_database_'),
    ],
    // ...
],

Now you can query the real keys in redis-cli:

redis-cli

# include the prefix when listing / inspecting keys
KEYS laravel_database_queues:*
LLEN laravel_database_queues:default
ZRANGE laravel_database_queues:default:reserved 0 -1

# don't know the prefix? match everything that ends in queues:*
KEYS *queues:*

Good to know

  • Artisan commands (queue:clear, Horizon, etc.) add the prefix for you — you only deal with it when hitting Redis directly.
  • Horizon may use its own connection/prefix; check config('horizon.prefix') (defaults to horizon:) for Horizon's own keys.
  • Set REDIS_PREFIX in .env to override it.
